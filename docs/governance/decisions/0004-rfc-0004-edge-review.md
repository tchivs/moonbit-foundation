# Edge Review Record: RFC 0004 (mb-font)

> **Historical record.** The edge-review process this file records was a mandatory precondition of the `sole-project-owner-bootstrap` acceptance route, which was removed from the [RFC process](../../rfcs/README.md) on 2026-07-23. This edge review is retained as a genuine architectural review of RFC 0004, but it is **no longer a mandatory acceptance precondition** — no acceptance route exists in the simplified process. The "Acceptance-route note" below should be read as a description of the now-removed machinery, not a live requirement.

- **RFC status at review time:** Proposed
- **Review date:** 2026-07-23
- **Reviewer:** edge-review (recorded under rfc-process §8)
- **Effect of this record:** Records the two mandatory edge reviews. Does NOT advance RFC 0004; it remains Proposed until an acceptance route is satisfied.

## Edge review results

- `EDGE-GOV-01-UNCLASSIFIED`: **Completed. Disposition: no-omission-found.** Scope reviewed: module identity `tchivs/mb-font` in the Document and Scene Layer; the single downward dependency edge (`mb-font -> mb-core`); explicitly rejected edges to `mb-color`, `mb-image`, and `mb-canvas` (with stated reasons); the font-vs-rasterization boundary (§3.1: binary-to-outline is font work, outline-to-pixel is canvas work; hinting redefined as a future *bounded outline adjustment*, never a pixel op); the font-vs-text boundary (§3.2: font owns per-glyph geometry and metrics; text owns string-to-positioned-glyph mapping); the deferred font-files-vs-selection/registry split (§3.3); the portable-core vs native-leaf seam (§5); the v0.x in-scope subset (TrueType `glyf` + CFF1, required tables, basic `kern`); binding deferrals (variable fonts, CFF2, color fonts, hinting VM, etc.); the deferred-layer rule (RFC 0001 §6.4); and the accepted-RFC gate. Result: no material architectural omission or blocking objection found — every named boundary is either explicitly drawn, explicitly deferred, or explicitly rejected, and the single downward edge is consistent with the layered model. Non-blocking observations:
  1. **Hinting determinism.** v0.x is deterministic *because* hinting is deferred (unhinted outlines = design-space geometry). However, §7.2's future "bounded outline adjustment" does not bind any future hinting RFC to preserve cross-target determinism. Recommend: any future hinting RFC must explicitly re-establish the determinism contract, as it is load-bearing for the layer's portability guarantee. Non-blocking for v0.x.
  2. **CFF (cubic) vs `glyf` (quadratic) public representation.** Both are in scope (§6.1), but the public outline representation is undecided (§11 defers quadratic-vs-cubic and fixed-vs-float). If a quadratic-only public type is chosen, CFF cubic charstrings will need cubic-to-quadratic conversion at extraction — an outline transformation not currently named as owned or excluded. Borderline implicit; treated as a §11-deferred internal detail that does not touch the public boundary, so non-blocking.
  3. **Same-layer composition with `mb-svg`/`mb-pdf`.** Rendering font outlines into an SVG path or PDF (same-layer composition, no dependency edge) is not explicitly named. It is allowed under the same-layer composition rule, but is currently implicit. Minor.
  4. **Runtime font licensing / OS/2 embedding bits.** These are implicitly out of scope — almost certainly the correct call, but unstated in the charter.

- `EDGE-GOV-02-UNCLASSIFIED`: **Completed. Disposition: no-omission-found.** Scope reviewed: RFC 0004 status is Proposed; the header is honest (no acceptance route, approvals, or evidence claimed, and review is recorded as open); the ledger shows only the `-> Draft -> Proposed` transition; the RFC fails closed (no unclaimed authority or transition); and the header, ledger, and index agree. Result: no omitted transition or authority case; no blocking objection. Non-blocking forward-looking finding: the sole-project-owner-bootstrap route (§4.3 / Decision 0001) is scoped "Applies to: RFC 0001 only" and its preauthorization was consumed by RFC 0001 under `AUTH-NO-LATER-APPROVAL`, so RFC 0004 cannot reuse it. Advancing RFC 0004 from Proposed requires either (a) the project-lead public-review route (§4.2: a ≥7-day public review window followed by project-lead approval) or (b) the maintainer route (§4.1, currently unavailable). If the sole-owner route were ever re-opened for general use, RFC 0004 would still require its own edge-review-results — Decision 0001's results are RFC-0001-scoped and do not transfer. This is a live authority gap, not a Proposed-header defect.
- Unresolved blocking objections: **none**.

## Acceptance-route note (does not advance the RFC)

This record satisfies the rfc-process §8 requirement that both mandatory edge reviews (EDGE-GOV-01 and EDGE-GOV-02) be completed before an RFC may be accepted. It is recorded while RFC 0004 is Proposed and does not, by itself, advance the RFC: no acceptance route has been satisfied. EDGE-GOV-01 found no material architectural omission — the layer placement, the single downward edge to `mb-core`, the explicitly rejected sibling edges, and the font-vs-rasterization / font-vs-text / portable-core-vs-native-leaf boundaries are all either drawn, deferred, or rejected with reasons; the four recorded observations are non-blocking. EDGE-GOV-02 found no omitted lifecycle transition and no defect in the Proposed header or ledger, but records a live, forward-looking authority gap: RFC 0004 cannot inherit Decision 0001's sole-project-owner preauthorization, which was scoped to RFC 0001 and already consumed. Acceptance therefore still requires a §4.2 project-lead public review (≥7-day window plus project-lead approval) or, if it becomes available, the §4.1 maintainer route. Until one of those routes is completed, RFC 0004 remains Proposed; the non-blocking observations above may be addressed in a future revision but are not conditions for advancing.

# Edge Review Record: RFC 0006 (mb-pdf)

> **Historical record.** The edge-review process this file records was a mandatory precondition of the `sole-project-owner-bootstrap` acceptance route, which was removed from the [RFC process](../../rfcs/README.md) on 2026-07-23. This edge review is retained because it is a genuine architectural review of RFC 0006 — it found the object-stream (ObjStm) scope gap and prompted the §6.1 fix that brought ObjStm explicitly in scope. It is **no longer a mandatory acceptance precondition** — no acceptance route exists in the simplified process. The "Acceptance-route note" below should be read as a description of the now-removed machinery, not a live requirement.

- **RFC:** 0006 — mb-pdf Charter
- **RFC status at review time:** Proposed
- **Review date:** 2026-07-23
- **Reviewer:** edge-review (recorded under rfc-process §8)
- **Effect of this record:** Records the two mandatory edge reviews. Does NOT advance RFC 0006; it remains Proposed until an acceptance route is satisfied.

## Edge review results

- `EDGE-GOV-01-UNCLASSIFIED`: **Completed. Disposition: no-omission-found (after charter revision).** Scope reviewed: module identity (tchivs/mb-pdf in Document and Scene Layer); three downward edges (mb-pdf -> {mb-image, mb-color, mb-core}) mirroring accepted mb-image edges; container-vs-renderer boundary (§3.1 parses/serializes structure, does NOT rasterize; §3.2–§3.4 cuts); the stated cut "PDF owns structure and serialization; rendering is a separate concern"; codec delegation to mb-image (§3.3); font-resource wrapper (§3.4, extracts TrueType/CFF streams for mb-font, non-edge); DEFLATE reuse from mb-image/png (§5, §7.2, §8); color via mb-color; portable seam (§5); v0.x subset with binding deferral; accepted-RFC gate. Result: no material architectural omission or blocking objection found AFTER the ObjStm charter revision. Non-blocking observations:
  1. **ObjStm scope unclassified — NOW RESOLVED by charter revision.** §6.1 previously listed "cross-reference table and stream parsing" as IN but said nothing about object streams (ObjStm, PDF 1.5+), which require FlateDecode of a compressed object container and have their own parse-order/security considerations, while §7.2 already treated "crafted object streams" as a known hostile-input category — an internal inconsistency. §6.1 Structure bullet now explicitly includes "compressed object streams (ObjStm, PDF 1.5+)", resolved through the same bounded FlateDecode path shared with mb-image's PNG DEFLATE discipline, in scope for object resolution (not deferred), because real-world PDF 1.5+ documents store the majority of objects in ObjStm and a parser that cannot resolve them is not interchange-useful. Consistent with §7.2.
  2. **Decode-vs-encode asymmetry implicit.** §8 rejects duplicating JPEG/PNG/DEFLATE decode and §3.3 says PDF orchestrates filter chains and delegates codec decode/encode to mb-image. But whether ENCODE of a PDF embedding a JPEG/PNG is symmetric with decode (does mb-pdf emit DCTDecode by calling an mb-image JPEG encoder, or only wrap pre-encoded bytes?) is never explicitly stated — the filter-chain cut is drawn only on the decode side. Non-blocking; recommend clarifying the encode-side filter contract.
  3. **mb-font reverse-edge risk under extraction implicit.** §3.4/§4.2 say PDF extracts embedded TrueType/CFF streams "for mb-font" and owns ToUnicode/CID mapping. Direction is mb-pdf -> mb-font (fine), but the ToUnicode/CID mapping data living in mb-pdf could pressure a reverse edge if mb-font ever needs to feed back on an encode path. Minor; data-flow direction not explicitly stated.
  4. **ICBased profile stream ownership vs mb-color.** §4.1 says PDF handles "ICC profile streams via mb-color's profile seams"; RFC 0001 §6.2 says mb-color owns bounded profile identity/metadata seams. Who PARSES the ICC stream bytes (PDF as filter-chain owner, or mb-color as profile owner) is unstated. Minor-ambiguous.

- `EDGE-GOV-02-UNCLASSIFIED`: **Completed. Disposition: no-omission-found.** Scope reviewed: status Proposed; header honest (no route/approvals/evidence, review open); ledger trace — -> Draft -> Proposed; fails closed; header/ledger/index agree on Proposed. Result: no omitted transition/authority case; no blocking objection. Non-blocking forward-looking finding: sole-project-owner-bootstrap (§4.3 / Decision 0001) was scoped "RFC 0001 only" and its preauthorization has been consumed (AUTH-NO-LATER-APPROVAL), so RFC 0006 cannot reuse it; the header correctly claims no route. To advance, RFC 0006 must satisfy either (a) project-lead public-review (§4.2: ≥7-day window + project-lead approval) or (b) maintainer route (§4.1, currently unavailable — one maintainer). The RFC does not preemptively warn against a future agent misapplying Decision 0001's RFC-0001-only preauthorization to RFC 0006; the honest "no route claimed" header guards against it, but the gap is worth noting. Not a Proposed-header defect; a live authority gap for the acceptance phase.

- Unresolved blocking objections: **none**.

## Charter revisions prompted by this review

- RFC 0006 §6.1 Structure bullet: object streams (ObjStm, PDF 1.5+) now explicitly in scope for object resolution, via the shared bounded FlateDecode path; consistent with §7.2 hostile-input handling. This closes the internal inconsistency under which §6.1 omitted ObjStm while §7.2 already classified crafted object streams as hostile input. The resolution routes ObjStm through the same bounded FlateDecode discipline already accepted for mb-image's PNG DEFLATE, keeping it a reuse rather than a new downward edge, and treats object-stream resolution as in scope (not deferred) because PDF 1.5+ interchange documents store the majority of their objects in ObjStm.

## Acceptance-route note (does not advance the RFC)

This record captures the two edge reviews required by rfc-process §8 prior to acceptance. It does not constitute acceptance and does not advance RFC 0006; the RFC remains Proposed. The architectural review (EDGE-GOV-01) found no material omission once the ObjStm scope was closed by charter revision; the lifecycle review (EDGE-GOV-02) found no omitted transition or authority case but flagged a live acceptance-route gap: RFC 0006 cannot inherit Decision 0001's sole-project-owner preauthorization, which was scoped to RFC 0001 only and has been consumed. To advance from Proposed, RFC 0006 must independently satisfy either the project-lead public-review route (§4.2: a public review window of at least seven days plus project-lead approval) or the maintainer route (§4.1, currently unavailable with only one maintainer). The four non-blocking architectural observations (encode-side filter contract, mb-font reverse-edge risk, ICC profile stream ownership, and the now-resolved ObjStm classification) are recorded for the author's consideration and do not gate acceptance.

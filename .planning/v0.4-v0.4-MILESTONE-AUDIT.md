# v0.4 Portable Image Interchange — Milestone Audit

## Verdict

**GAPS** — the six v0.4 QOI requirements and the Phase 13 → 14 → 15 public
flow are demonstrably wired and pass on all four supported targets. The
milestone nevertheless contains out-of-scope registry/release changes and a
current foundation-policy regression introduced by the new example workspace
member. It must not be treated as a clean milestone pass until those unrelated
changes are reconciled.

## Gaps

### BLOCKER — Phase 15 workspace integration breaks the existing foundation-policy gate

`examples/qoi-portable` was correctly added to `moon.work` so the public
example can run, but `scripts/quality/Assert-Policy.ps1` still expects only
the three modules plus `examples/ppm-portable` and `examples/ppm-native-cli`.
Dot-sourcing the policy script and running
`Assert-FoundationPolicy -PolicyPath policy/foundation.json` fails with:

```text
moon.work members count mismatch: expected 5, got 6.
```

This is a cross-phase integration break between the Phase 15 workspace member
and the existing foundation policy/Required quality gate. The v0.4 plan did
not update the policy contract, so the repository cannot presently claim that
its established policy gate remains coherent after adding the public example.

### WARNING — out-of-scope registry artifacts were reintroduced during Phase 15

Commit `9d017a6` (`docs(15): wire qoi example workspace`) added four files under
`release/registry/`: authority-observation schema/data and capability-matrix
schema/data. They are not among the Phase 15 plan's five allowed implementation
files and conflict with its explicit prohibition on registry work. They do not
participate in the QOI runtime path, but they violate the milestone scope.

### WARNING — out-of-scope release-governance automation was removed during v0.4

Commit `2518370` (`refactor(quality): remove v0.1/v0.2 release-governance test
scripts`) changed `Assert-Policy.ps1` and `Invoke-MoonQuality.ps1` and deleted
nine release/registry qualification scripts plus their shared library (2,965
deleted lines). v0.4 explicitly defers release automation; these changes are
unrelated to QOI decode/encode/public-consumer delivery and were not planned by
Phases 13–15. The QOI tests still pass, but this scope breach needs a separate,
reviewed change or a deliberate rollback.

## Scope check

The actual QOI package remains a pure-MoonBit eager codec over existing
forward-only `Reader`/`Writer` contracts. There is no QOI FFI/native binding,
registry lookup, streaming state API, benchmark, PNG/DEFLATE implementation,
or release behavior in `modules/mb-image/qoi` or `examples/qoi-portable`.

The two scope findings above concern unrelated files committed in the v0.4
timeline, not an unwired QOI codec path.

## Required follow-up

1. Reconcile `policy/foundation.json` / `Assert-Policy.ps1` with the approved
   `examples/qoi-portable` workspace member, then rerun the foundation policy
   assertion and the required quality lane as appropriate.
2. Move the four `release/registry/` additions and commit `2518370` into a
   separately reviewed release-governance change, or revert them, before
   declaring v0.4 scope-clean.

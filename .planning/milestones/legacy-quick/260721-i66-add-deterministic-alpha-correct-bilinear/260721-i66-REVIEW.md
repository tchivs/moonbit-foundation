## Critical Issues

### CR-01 — Quantized-zero alpha is emitted with nonzero RGB

**Classification:** BLOCKER  
**File:** `modules/mb-image/ops/processing.mbt:113-128`  
**Issue:** `store_linear_premultiplied` canonicalizes only when the normalized floating-point alpha is exactly `0.0`. Bilinear interpolation can produce a positive alpha that quantizes to encoded alpha byte `0`. The function then unpremultiplies and writes RGB normally, producing an invalid noncanonical transparent pixel.

For example, resize `[255,0,0,1] [0,0,0,0]` from `2x1` to `3x1`. At destination `x=1`, the documented coordinate gives the red tap a one-third weight: alpha is `1 / (3 * 255)`, which rounds to encoded alpha `0`, while unpremultiplication restores red to `255`. The result is `[255,0,0,0]`, contradicting the documented guarantee that zero-alpha output is `[0,0,0,0]` and reintroducing the transparent-color fringe the operation is meant to prevent.

**Fix:** Quantize alpha before deciding canonicalization, then write transparent black whenever that encoded byte is zero. Only unpremultiply and quantize RGB for a nonzero encoded alpha. For example:

```mbt
let alpha_byte = @quantize.quantize_alpha(value.alpha()).value()
if alpha_byte == b'\x00' {
  for channel = 0UL; channel < 4UL; channel = channel + 1UL {
    match view.set_byte(x, y, channel, b'\x00') {
      Ok(_) => ()
      Err(error) => return Err(error)
    }
  }
  return Ok(())
}
let straight = match @alpha.unpremultiply_normalized(value) {
  Ok(value) => value
  Err(error) => return Err(error)
}
// Quantize RGB as today, and store alpha_byte as channel 3.
```

### WR-01 — Regression vectors omit the alpha-quantization boundary

**Classification:** WARNING  
**File:** `modules/mb-image/ops/resize_convert_test.mbt:250-285`  
**Issue:** The transparent-fringe test covers exactly zero alpha and a large interpolated alpha (`170`), so it passes despite CR-01. It does not exercise the decisive case where a positive interpolated alpha rounds down to byte zero; the README proof repeats the same gap at `modules/mb-image/README.mbt.md:249-273`.

**Fix:** Add a `2x1 -> 3x1` straight-RGBA test with `[255,0,0,1] [0,0,0,0]` and assert that output pixel `x=1` is exactly `[0,0,0,0]` on every portable target. Keep this vector in the README proof as well if that proof is intended to substantiate the canonicalization contract.

# v0.11 PNG Dynamic Huffman Compression — Research

**Researched:** 2026-07-22  
**Recommendation confidence:** HIGH for repository integration; MEDIUM for the RFC details independently rechecked this session.

## Scope and Recommendation

Implement one new, explicit `PngCompressionStrategy::DynamicOrFixedOrStored` route. It must select Dynamic only when a complete, exactly planned PNG is **strictly smaller** than the byte sequence the unchanged `FixedOrStored` route would select; otherwise it must emit that existing Fixed-or-Stored winner. `Stored`, legacy `PngEncoder::new()`, legacy `PngChunkEncoder::new(...)`, and existing `FixedOrStored` must retain their current bytes. [VERIFIED: codebase: `modules/mb-image/png/{png.mbt,encode.mbt,stream_encode.mbt}`; `.planning/milestones/v0.10-MILESTONE-AUDIT.md`]

This is the highest-value safe slice because the repository already has (1) a public additive strategy seam, (2) a single atomic preflight shared by eager and caller-buffered construction, (3) an acknowledgement-safe scalar DEFLATE emitter, and (4) a decoder-side dynamic-tree parser/canonical Huffman builder. The addition should reuse those seams rather than create a second PNG formatter, token/output staging path, or a new compression package. [VERIFIED: codebase: `modules/mb-image/png/{png.mbt,encode.mbt,stream_encode.mbt,deflate_huffman.mbt,deflate_inflate.mbt}`]

### Hard scope fences

- Keep filter-None scanlines and the current deterministic matcher limited to distances 1 through 4; do not add adaptive filtering, a 32 KiB dictionary, or broader matching. [VERIFIED: codebase: `modules/mb-image/png/encode.mbt`; `.planning/milestones/v0.10-REQUIREMENTS.md`]
- Do not use FFI, host stream adapters, external packages, release/registry work, CI changes, APNG, colour work, or metadata expansion. [VERIFIED: codebase: `AGENTS.md`; `.planning/milestones/v0.10-ROADMAP.md`]
- Do not alter the one-IDAT framing, zlib wrapper, IDAT CRC, Adler-32 behaviour, eager writer lifecycle, caller-lease ownership, or sticky terminal contract. [VERIFIED: codebase: `modules/mb-image/png/{encode.mbt,stream_encode.mbt}`; `.planning/milestones/v0.10-MILESTONE-AUDIT.md`]

## Technical Design

### Candidate selection and bounded preflight

1. Compute the present Stored plan exactly, then compute the current Fixed plan exactly and preserve its existing `fixed.total_length <= stored.total_length` choice unchanged. [VERIFIED: codebase: `modules/mb-image/png/encode.mbt`]
2. Only for `DynamicOrFixedOrStored`, run the same distance-1-through-4 token walk to collect literal/length and distance frequencies, construct a canonical Dynamic codebook, and count the exact dynamic header, token, EOB, padding, zlib, IDAT, and complete-PNG bytes. [CITED: https://datatracker.ietf.org/doc/html/rfc1951] [VERIFIED: codebase: `modules/mb-image/png/{encode.mbt,deflate_inflate.mbt}`]
3. Select `Dynamic` only if its complete `total_length` is strictly less than the already-selected Fixed-or-Stored total; on a tie, retain the existing Fixed-or-Stored representation. This makes the new strategy never larger while avoiding a needless representation change. [ASSUMED: selection policy; planner must retain this explicit tie rule]
4. Perform max-output, max-work, and the single budget charge against the selected exact plan before either public adapter creates an output machine. [VERIFIED: codebase: `modules/mb-image/png/encode.mbt`; `.planning/milestones/v0.10-phases/33-fixed-or-stored-png-planning-and-emission/33-VERIFICATION.md`]

Use an ordinary deterministic Huffman merge builder only when its resulting code lengths are all at most 15; if it cannot produce a valid bounded codebook, treat Dynamic as unavailable for that input and fall back to the existing Fixed-or-Stored winner. Do **not** add a length-limited/package-merge optimizer in this milestone. DEFLATE allows compressors to further limit valid ranges, while its code lengths are capped at 15. [CITED: https://datatracker.ietf.org/doc/html/rfc1951] [ASSUMED: ordinary-builder fallback is the safest v0.11 policy]

### Fixed-size dynamic plan

`PngDynamicPlan` should contain only fixed-upper-bound, image-independent planning facts: literal/length (286), distance (30), and code-length (19) code lengths/codes; `HLIT`, `HDIST`, `HCLEN`; exact DEFLATE bits/bytes; matcher/planning work; IDAT/PNG lengths; and no more state than is needed to deterministically regenerate the code-length RLE header. It must not retain scanlines, all tokens, compressed bytes, a caller view/lease, or a 32 KiB history buffer. [CITED: https://datatracker.ietf.org/doc/html/rfc1951] [VERIFIED: codebase: `modules/mb-image/png/{encode.mbt,stream_encode.mbt}`]

The Dynamic block is a single final block with BFINAL/BTYPE bits `101` in LSB-first accumulator order (`BTYPE=10`). Emit the RFC header in this order: HLIT, HDIST, HCLEN; code-length alphabet lengths in the required permutation; then the combined literal/length plus distance length sequence RLE-coded with symbols 16/17/18; then encoded data and EOB. The existing decoder implements precisely these semantics and is the local executable reference. [CITED: https://datatracker.ietf.org/doc/html/rfc1951] [VERIFIED: codebase: `modules/mb-image/png/deflate_inflate.mbt`]

For an all-literal token stream, emit a legal one-bit distance singleton even when it is unused; for the code-length alphabet, deterministically include a second one-bit leaf if one would otherwise be incomplete. Existing `_png_huffman_new` validates the first exception for distance trees but rejects generic incomplete alphabets, so the encoder must generate trees it can decode. [VERIFIED: codebase: `modules/mb-image/png/deflate_huffman.mbt`] [CITED: https://datatracker.ietf.org/doc/html/rfc1951]

### Resumable emission and work accounting

Extend the private machine with `PngDeflatePlan::Dynamic(PngDynamicPlan)` and a `PngDynamicState`. The state needs scalar cursors for header RLE, raw scanline/token replay, bit accumulator/count, Adler, and replay-work; its preview result is committed only by `acknowledge`. Continue using the present machine for PNG framing and CRC ranges. [VERIFIED: codebase: `modules/mb-image/png/stream_encode.mbt`]

The selected Dynamic work ledger must include bounded planning work, the matcher walk performed for preflight, replay matcher work, and output work; record the exact definition in code and test its exact/one-less boundary. At final EOB, assert replay work and emitted bit count equal the plan; a drift caused by mutable source data must become the same style of sticky terminal failure already used by Fixed replay, never a length/progress mismatch. [VERIFIED: codebase: `modules/mb-image/png/{encode.mbt,stream_encode.mbt}`] [ASSUMED: exact dynamic ledger formula needs implementation-level confirmation]

The outer zlib/PNG handling remains unchanged: method-8 zlib header without FDICT, DEFLATE payload, big-endian Adler-32 of uncompressed filtered bytes, and one IDAT whose data is the zlib stream. PNG IDAT boundaries have no required relationship to scanlines or DEFLATE blocks, so retaining the current single-IDAT output is valid. [CITED: https://datatracker.ietf.org/doc/html/rfc1950] [CITED: https://www.w3.org/TR/png-3/]

### Phase 37 — Four-target dynamic compression evidence

Add a generated, in-memory periodic five-symbol RGB8 and straight-RGBA8 corpus large enough to make the current 1–4-distance matcher produce literals rather than matches. For each case: compare explicit Stored, unchanged FixedOrStored, and DynamicOrFixedOrStored; require Dynamic to be strictly smaller than FixedOrStored, repeated eager determinism, hostile-schedule chunk parity, BTYPE=10 selection, and complete-input public decode to every source component. Run outline-before-filter checks in isolated target directories on js, wasm, wasm-gc, and native, then the complete PNG package suite. | Risk | Prevention / executable evidence |
|---|---|
| Dynamic header has wrong permutation, RLE, canonical reversal, or BTYPE bits. | White-box exact header tests plus public complete-input decode; use existing `_png_dynamic_trees` and `_png_huffman_new` as the local reference. [VERIFIED: codebase: `modules/mb-image/png/{deflate_inflate.mbt,deflate_huffman.mbt}`] [CITED: https://datatracker.ietf.org/doc/html/rfc1951] |
